# ProofAtlas Project Summary and Next Plan

Date: 2026-06-20

## One-Sentence Summary

ProofAtlas aims to turn Lean/mathlib proof knowledge into a knowledge graph, then use that graph as the retrieval and recommendation engine for theorem proving assistance.

## Core Goals

ProofAtlas has two main goals.

First, it builds and presents a Lean/mathlib proof knowledge graph. The graph organizes proof-related objects such as:

- theorems and lemmas,
- proof states,
- premises and definitions,
- files and modules,
- tactic steps,
- proof techniques,
- proof patterns.

This graph is intended to make relationships in mathematical proof knowledge visible, including theorem-to-premise usage, theorem-to-theorem similarity, proof-state-to-proof-pattern links, and proof-technique evidence.

Second, ProofAtlas uses the knowledge graph to build a proof assistance system. Given a new theorem or proof state, the system should retrieve, rank, and explain useful proof guidance from existing Lean/mathlib proof knowledge.

## Target User Workflow

The target workflow is:

```text
Given a new Lean theorem or proof state
-> parse and encode the query
-> retrieve relevant knowledge from the Lean/mathlib proof KG
-> rank and explain the retrieved results
-> return proof guidance that helps users or provers find a proof path
```

For a new theorem, the system should return:

- Relevant lemmas and premises: likely useful theorems, lemmas, definitions, and facts.
- Similar theorems: mathlib theorems with similar statement structure, dependency patterns, or proof shape.
- Likely proof techniques: possible tactics or strategies such as induction, rewriting, simplification, case split, contradiction, or algebraic manipulation.
- Related proof states and proof patterns: historical goals and contexts where similar proof patterns succeeded.
- Difficulty profile: estimated proof complexity based on dependency count, goal structure, namespace/domain overlap, and negative candidate hardness.
- Premise ranking explanations: reasons why each premise was recommended, such as statement similarity, shared symbols, namespace overlap, graph evidence, or prior use in similar proof states.

## Demo Page Scope

The final homepage or demo page should present two main experiences.

### 1. Knowledge Graph Visualization

The page should visualize how Lean/mathlib proof knowledge is organized into graph nodes and edges. It should show relationships among theorems, premises, proof states, files/modules, tactic steps, and proof techniques.

The goal is to let users quickly understand how theorem proving knowledge is represented as a graph rather than as isolated proof scripts.

### 2. New Theorem Proof Guidance Demo

The page should let users input or select a theorem and then display proof guidance generated by the system.

The output should include:

- relevant premises,
- similar theorems,
- likely proof techniques,
- related proof states and proof patterns,
- difficulty profile,
- ranking explanations.

## Current Status

ProofAtlas has moved beyond a basic proof-state retrieval prototype. The current implementation includes an end-to-end theorem-guidance MVP over processed LeanRank/mathlib artifacts.

Completed components include:

- KG nodes for `Theorem`, `ProofState`, `Premise`, `FileModule`, `TacticStep`, and `ProofTechnique`.
- Theorem-level premise invocation edges and proof-state-level positive/negative premise evidence.
- Embeddings for proof states, premises, and theorems.
- Persistent nearest-neighbor indexes with sklearn as the default backend and optional hnswlib, FAISS, and LanceDB backends.
- Direct similar-proof-state retrieval from historical proof-state embeddings.
- Index benchmark reports for latency, build time, and top-k overlap against exact cosine retrieval.
- Retrieval for out-of-KG theorem or proof-state text.
- Structured query parsing for Lean theorem, lemma, and example snippets, including structured binders, conclusion symbols, operator symbols, sort/typeclass symbols, and alpha-normalized goal text.
- Theorem query parse coverage report included in refresh dashboard parsing diagnostics.
- Unified theorem guidance retrieval that returns premises, similar theorems, similar proof states, proof techniques, proof patterns, graph evidence, difficulty signals, and ranking explanations.
- Optional Lean syntax diagnostics through `lake env lean` or `lean`.
- Initial proof-state extraction from Lean unsolved-goal diagnostics.
- Learned premise-ranker integration when a trained ranker artifact is available.
- Learned-ranker feature ablation reports for embedding, namespace/domain, proof-technique, difficulty, frequency, symbol-overlap, graph, and theorem-neighborhood signals.
- Theorem-level evaluation with Recall@k, MRR, MAP, nDCG, coverage counts, and case studies.
- Initial trained difficulty estimator signal with theorem-complexity targets derived from proof length, tactic count, premise count, and negative-candidate signals.
- Adaptive retrieval policy that expands premise candidate depth for harder queries before reranking.
- Difficulty estimator calibration bins, residual quantiles, and guidance-time uncertainty intervals.
- Static homepage sections for KG visualization and theorem-guidance examples.
- Optional FastAPI service with `/health` and theorem-guidance endpoints.
- API metrics, request IDs, structured logging hooks, optional startup readiness enforcement, and a deployment security/readiness review report.
- Deployment guide for static homepage, local API, readiness-gated server, notebook, and hosted demo modes.
- Corpus provenance manifest and expanded audit checks.
- Corpus data-supervision profile distinguishing synthetic demo rows and LeanRank proof-state rows.
- Artifact compatibility report for config, processed tables, embeddings, and index manifests.
- Artifact compatibility warnings when a corpus is internally consistent but not suitable for prediction-quality claims.
- Refresh dashboard for KG scale, domain coverage, retrieval quality, parsing coverage, index benchmarks, difficulty calibration, and compatibility gates.

The strongest working path today is:

```text
Given a new theorem statement or proof goal
-> build a structured query
-> optionally validate it with Lean
-> embed the query
-> retrieve and rerank train-side premises and similar theorems
-> attach graph evidence, proof-technique hints, proof-pattern evidence, difficulty signals, and explanations
-> return JSON guidance through CLI, Python API, or the optional web service
```

## Main Remaining Gaps

The current system is a strong MVP, but several gaps remain before it becomes a robust mathlib-scale proof assistance system.

### 1. Lean-Aware Parsing

The current parser handles common theorem and binder structure, parsed conclusion symbols, operator symbols, sort/typeclass symbols, and alpha-normalized goals, and it writes theorem query parse coverage diagnostics. It is still lightweight.

Next steps:

- Integrate a robust Lean parser, Lean server workflow, or tree-sitter-style syntax extractor.
- Extract namespaces, constants, binders, hypotheses, typeclass constraints, and conclusion structure more reliably through a real Lean parser/session.
- Normalize statements more completely so alpha-renaming, notation expansion, coercions, and local binder names do not dominate retrieval.
- Expand parsed query features in ranking and explanations beyond the current binder/conclusion-symbol signals.

### 2. Real Proof-State Extraction

Current proof-state extraction uses structured Lean `unsolved goals` diagnostic reports and processed LeanRank proof states, but it still lacks a full Lean server/session tactic-state stream.

Next steps:

- Replace diagnostic-text extraction with a full Lean server/session tactic-state stream when available.
- Map new proof states into the same representation used by historical proof states.
- Retrieve proof patterns from both theorem-level text and actual goal/context structure.
- Validate extraction diagnostics against real Lean server/session traces.

### 3. Scalable Retrieval

The current index layer supports local nearest-neighbor retrieval, optional hnswlib ANN indexing, optional FAISS indexing, optional LanceDB vector-store indexing, corpus-aware index manifests, and deterministic benchmark reports, but larger deployments need stronger scalability validation.

Next steps:

- Validate LanceDB and other production vector-store choices on larger mathlib-scale corpora.
- Keep sklearn as the lightweight default for tests and small demos.
- Extend benchmark coverage across larger corpora, more query distributions, and future production vector-store backends.

### 4. Stronger Learned Ranking

The current learned ranker is optional and feature-based, with feature-group ablation reporting over embedding, namespace/domain, proof-technique, difficulty, frequency, symbol-overlap, graph premise-degree, and theorem-neighborhood premise evidence.

Next steps:

- Train richer premise-ranking models beyond the current logistic feature reranker.
- Add deeper theorem-similarity, dependency-path, and graph-neighborhood signals into ranker training.
- Compare fixed scoring, GBDT/logistic reranking, cross-encoder reranking, and graph neural reranking.
- Extend ablations to richer graph evidence, theorem-neighborhood features, symbol overlap, and neural rerankers.

### 5. Better Difficulty Modeling

The current difficulty profile combines heuristics, historical priors, and an initial trained estimator with residual-based uncertainty diagnostics. Difficulty feature generation now includes a theorem-complexity target derived from proof length, tactic count, premise count, failed negative-candidate count, negative-candidate hardness, and proof-state difficulty. Guidance uses a pre-retrieval difficulty estimate to expand premise candidate depth for harder queries before reranking.

Next steps:

- Validate the current theorem-complexity target against stronger or external proof-complexity labels.
- Train richer theorem-level and proof-state-level estimators with dependency-depth and failed-proof-attempt features.
- Validate calibrated confidence and uncertainty against stronger proof-complexity labels.
- Validate and tune adaptive retrieval-depth policy against stronger proof-complexity labels, then extend difficulty estimates to proof-strategy recommendations.

### 6. Production-Ready API and Demo

The current FastAPI and homepage demo are useful for inspection and include initial operational hooks, including JSON request metrics, Prometheus-compatible metrics text export, request IDs, structured logging hooks, readiness-gated startup, and a generated deployment security/readiness review report. They still need production hardening.

Next steps:

- Connect the current metrics endpoints to persistent production monitoring and deployment log integration.
- Expand deployment-specific startup policy documentation.
- Improve frontend loading, empty, error, and slow-query states with clearer recovery actions.
- Expand deployment documentation with production-specific infrastructure examples once a target host is selected.
- Use the generated security review as a public-exposure gate, and replace deployment assertion flags with host-specific auth, rate-limit, network, TLS, and monitoring integration.

### 7. LeanRank-data Scale-Up

The current project scope uses `erbacher/LeanRank-data` as the dataset source. The pipeline includes a corpus manifest, data-supervision profile, artifact compatibility report, refresh dashboard, previous-snapshot refresh trend report, bounded refresh history, held-out test-set evaluation, premise-supervision report, and pipeline performance/timing report.

Next steps:

- Scale the LeanRank-data run and compare held-out test-set metrics across runs.
- Tune index and embedding backends based on pipeline timing and benchmark reports.
- Expand artifact compatibility tests across real mathlib refreshes and historical artifact versions.
- Validate refresh trend/history comparisons across repeated real mathlib refreshes and choose durable storage for long-running refresh history.

## Recommended Next Plan

The next development phase should focus on:

1. Improving Lean-aware theorem parsing and real proof-state extraction.
2. Adding scalable ANN or vector-store retrieval with benchmark reports.
3. Training a stronger premise reranker using graph and proof-state features.
4. Turning difficulty estimation into an evaluated prediction task.
5. Hardening the FastAPI service and homepage demo for external review.
6. Automating full mathlib data refresh and provenance tracking.

## Final Direction

ProofAtlas should first make Lean/mathlib proof knowledge visible as a theorem-centered knowledge graph. It should then use that graph to retrieve and explain proof guidance for new theorem statements, eventually becoming a practical retrieval engine for proof assistance.
